# SafeFill 项目阶段总路线图

> 重要提醒：如果后续发生上下文压缩、会话恢复、模型切换或记忆不完整，继续工作前必须先读取本文件。不要凭记忆继续执行。

## 项目总目标

创建一个“SafeFill”，帮助老师、行政工作人员、科研人员处理重复填表任务。

核心流程：

1. 用户导入以前填过的表格。
2. 系统在本地提取候选个人资料。
3. 用户确认后，资料才保存到本地隔离资料库 `vault\profile.json`。
4. 用户导入新的表格。
5. 系统根据本地资料库生成自动填写草稿。
6. 用户在本地网页中检查、修改、补充未知信息。
7. 用户点击“保存检查结果”后，系统生成正式 `review_result`。
8. 最后导出 Word、Excel 等目标格式文件。
9. 完成后可选运行 Cleaner 预览清理临时隐私文件。

## 总安全原则

1. API-first 模式，主流程使用大模型 API。
2. API 是唯一允许联网的接口。
3. API 默认关闭。
4. 每次 API 调用前，必须展示将发送的内容，并等待用户确认。
5. 不得自动上传身份证号、手机号、住址、证件照、完整原始表格。
6. 不得自动联网搜索个人信息。
7. 不得自动提交申报系统。
8. 不得修改用户原始文件。
9. 所有草稿必须输出到 `D:\SafeFill\draft_outputs\`。
10. 所有最终文件必须输出到 `D:\SafeFill\final_outputs\`。
11. 用户补充的信息默认不保存到资料库，只有用户明确勾选后才保存。
12. 不得读取 `D:\SafeFill\` 之外的私人文件。

## 第 0 阶段：需求边界和安全设计

目标：
先确定这个工具做什么、不做什么，明确安全边界。

产物：

1. `D:\SafeFill\SKILL.md`
2. `D:\SafeFill\docs\需求说明.md`
3. `D:\SafeFill\docs\安全设计.md`
4. `D:\SafeFill\docs\数据字段清单.md`
5. `D:\SafeFill\docs\第0阶段验收清单.md`

状态：
已完成。

核心结论：

1. 第一版只支持 `.docx` 和 `.xlsx`。
2. PDF 放到第二版。
3. 扫描版 PDF 和图片表格放到第三版。
4. 不自动联网搜索个人信息。
5. 不自动提交申报系统。
6. 不自动修改原始文件。
7. 不偷偷把完整资料发给 API。

## 第 1 阶段：本地资料库初始化

目标：
让用户把旧表格放入 `input_forms`，系统从旧 `.docx` / `.xlsx` 中提取候选信息，但不自动保存。用户确认后，才写入 `vault` 本地资料库。

主要功能：

1. 创建 `vault`、`input_forms`、`candidate_reviews`、`logs` 等目录。
2. 创建 `profile_template.json`。
3. 编写 `extract_candidates.py`。
4. 从旧 `.docx` / `.xlsx` 中提取候选字段。
5. 生成 `candidate_review_时间戳.md` 和 `candidate_review_时间戳.json`。
6. 在 Markdown 检查文件中脱敏显示身份证号、手机号、地址。
7. 生成 `latest_candidate.json`，指向最新候选结果。
8. 编写 `save_confirmed_profile.py`，让用户确认后把资料保存为唯一正式资料库 `vault\profile.json`。
9. 初版不支持多人物资料库，不使用 `person_001.json`、`person_002.json` 作为正式流程。

禁止事项：

1. 不做自动填新表。
2. 不做网页。
3. 不处理 PDF。
4. 不处理扫描件 OCR。
5. API 发送由用户配置和确认。
6. 不联网。
7. 不自动保存敏感信息。
8. 不覆盖已有用户资料。

## 第 2 阶段：新表格识别和自动填写草稿

目标：
用户上传一个新的 `.docx` 或 `.xlsx` 表格，系统读取表格结构，识别可填写字段，根据 `vault` 中已确认资料生成填写草稿。

主要功能：

1. 从 `input_forms` 或 `new_forms` 读取新表格。
2. 识别表格字段，例如姓名、性别、单位、职称、手机号等。
3. 与本地资料库字段匹配。
4. 生成填写计划 `filling_plan.json`。
5. 生成自动填写草稿文件。
6. 草稿输出到 `draft_outputs`。
7. 不覆盖原始文件。
8. 记录已填写字段、未知字段、需要复核字段。

禁止事项：

1. 不自动保存新信息进资料库。
2. 不直接输出最终版。
3. 不修改原始表格。
4. API 发送由用户配置和确认。
5. 不处理 PDF 和扫描件。

## 第 3 阶段：本地网页检查和补充

目标：
创建一个只在本地运行的网页，让用户检查系统填了什么、哪里不确定、哪里需要补充。

网页地址：

`http://127.0.0.1:8787`

主要功能：

1. 展示自动填写字段。
2. 展示字段来源。
3. 展示可信度。
4. 展示未知字段。
5. 允许用户修改自动填写内容。
6. 允许用户补充空缺内容。
7. 每个补充字段旁边提供选项：“是否保存到资料库”。
8. 用户确认后，生成最终填写数据。
9. 所有网页资源必须本地加载。

安全要求：

1. 只能绑定 `127.0.0.1`。
2. 不加载 CDN。
3. 不加载外部字体。
4. 不加载远程图片。
5. 不使用外部 JS。
6. 不联网。
7. 不自动保存用户补充内容。

## 第 4 阶段：按原格式导出

目标：
根据用户确认后的内容，尽量保持原表格格式，导出最终文件。

第一版导出：

1. `.docx` 保留 Word 表格格式。
2. `.xlsx` 保留 Excel 单元格格式、边框、合并单元格、字体。

第二版导出：

1. 支持可填写 PDF 表单。
2. 普通 PDF 只做预览和提示，不强行写入。

第三版导出：

1. 支持扫描 PDF。
2. 支持图片表格。
3. 支持 OCR 识别。
4. 支持更复杂的版式还原。

输出目录：

`D:\SafeFill\final_outputs\`

禁止事项：

1. 不覆盖原始文件。
2. 不把草稿当最终版。
3. 不破坏原始格式。
4. 导出失败时不得删除草稿或原始文件。

## 第 5 阶段：API 辅助理解字段

目标：
在本地规则无法判断字段含义时，允许用户选择是否调用 API 辅助判断。

API 只能做：

1. 判断字段含义。
2. 判断字段和资料库字段的对应关系。
3. 对复杂说明文字进行结构化理解。

API 禁止做：

1. 接收完整身份证号。
2. 接收完整手机号。
3. 接收证件照。
4. 接收完整原始表格。
5. 接收未脱敏个人资料。
6. 自动联网搜索个人信息。
7. 自动提交申报系统。

强制规则：

1. API 默认关闭。
2. 调用前展示发送内容。
3. 用户确认后才能发送。
4. 敏感字段默认脱敏。
5. API 调用写入本地日志。
6. 用户可以查看 API 调用记录。

## 第 6 阶段：封装成正式 Skill

目标：
把已经跑通的本地工具封装成 Codex / Claude Code 可以稳定调用的 Skill。

主要内容：

1. 完善 `SKILL.md`。
2. 明确调用入口。
3. 明确目录规则。
4. 明确安全规则。
5. 明确每阶段执行前必须检查的文件。
6. 明确每阶段执行后的验收标准。
7. 明确 Claude Code 不得自由扩大任务范围。
8. 明确 Claude Code 不得修改无关文件。
9. 明确 Claude Code 每阶段完成后必须汇报。

Skill 重点：

1. Skill 不应该直接存放大量业务代码。
2. Skill 应该告诉 Claude Code 如何安全调用本地程序。
3. Skill 应该限制 Claude Code 的行为边界。
4. Skill 应该要求所有敏感操作前必须用户确认。

## 第 7 阶段：测试、安全审计和真实样例验证

目标：
用模拟表格和脱敏样例测试整个流程，确认不会泄露资料、不会覆盖原文件、不会错误保存敏感信息。

测试内容：

1. 测试旧 `.docx` 信息提取。
2. 测试旧 `.xlsx` 信息提取。
3. 测试身份证号脱敏。
4. 测试手机号脱敏。
5. 测试地址脱敏。
6. 测试用户确认后保存资料。
7. 测试新表格自动填写草稿。
8. 测试本地网页检查。
9. 测试用户补充信息是否默认不保存。
10. 测试最终 `.docx` 导出。
11. 测试最终 `.xlsx` 导出。
12. 测试日志是否不包含完整敏感信息。
13. 测试断网情况下是否能运行。
14. 测试 API 默认关闭。

禁止事项：

1. 不使用真实身份证号。
2. 不使用真实手机号。
3. 不使用真实证件照。
4. 不使用真实未公开项目材料。
5. 不联网测试，除非用户明确要求 API 测试。
